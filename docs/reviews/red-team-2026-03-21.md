# Red Team Report — 2026-03-21

**Scope**: Exercise replay & navigation drawer design (#66) against existing codebase
**Auditors**: 3 parallel adversarial agents (state management, drawer patterns, edge cases)

---

## Broken Flows (app breaks)

### BF-1: `completeExercise` uses stale `optimalMoves` from render closure [HIGH]
- **File**: `hooks/use-exercise-progress.ts:69,81`
- **Description**: `completeExercise` computes stars using `currentExercise.optimalMoves` captured from the render closure, not from `prev.exerciseIndex` inside the `setProgress` updater. If `goToExercise(i)` is called and `completeExercise()` fires before React re-renders, stars are calculated against the wrong exercise's `optimalMoves`.
- **Steps to reproduce**: Navigate to exercise 3 via drawer → quickly tap target before re-render completes → stars computed against exercise 2's optimalMoves.
- **Fix**: Derive `optimalMoves` from `EXERCISES[piece][prev.exerciseIndex]` inside the updater function.

### BF-2: `autoResetTimer` captures stale closures, conflicts with drawer navigation [HIGH]
- **File**: `app/page.tsx:431-443`
- **Description**: The 1500ms `setTimeout` after exercise completion closes over `isLastExercise`, `advanceExercise`, and `pieceCompleted` from the render at call time. If the user opens the exercise drawer and navigates to a different exercise during these 1500ms, the timer fires and calls `advanceExercise()` — advancing one more exercise beyond where the user just navigated.
- **Steps to reproduce**: Complete exercise 2 → within 1.5s open drawer and tap exercise 1 → timer fires `advanceExercise()` → user lands on exercise 2 instead of staying on 1.
- **Fix**: Cancel `autoResetTimer` in `onNavigate` (the drawer's navigation callback), same as `resetBoard` does.

### BF-3: Board doesn't reset when navigating between exercises with same `startPos` [HIGH]
- **File**: `components/board.tsx:60-65`
- **Description**: Board's `useEffect` syncs state when `startPosition.file/.rank` change. But `rook-1` and `rook-2` both start at `pos(0,0)` — navigating between them via `goToExercise` doesn't trigger the effect. The piece stays wherever the user left it, `movesCount` isn't reset.
- **Affected exercises**: rook-1↔rook-2, bishop-1↔bishop-4 (both `pos(0,0)`).
- **Steps to reproduce**: Complete rook-1 (piece at h1) → navigate to rook-2 via stars bar → board shows piece at h1 instead of a1, target portal at a8 but piece is misplaced.
- **Fix**: The `onNavigate` callback must call `resetBoard()` (which increments `boardKey` forcing a full Board remount). Spec already specifies this but current `goToExercise` wiring in `page.tsx:764` does NOT call `resetBoard`.

## Dead Ends (user stuck)

### DE-1: Badge prompt re-fires on every replay of last exercise [MEDIUM]
- **File**: `app/page.tsx:419-428`
- **Description**: When replaying exercise 5 (last), the badge earned prompt fires if `newTotal >= BADGE_THRESHOLD && !hasClaimedBadge`. Since `hasClaimedBadge` comes from an on-chain query that may be `undefined` (pending), or `false` (not claimed yet), the prompt re-appears every time the user replays and wins the last exercise.
- **Trigger**: Replay exercise 5 with 10+ total stars and badge not yet claimed on-chain.
- **Fix**: Spec already identifies this — add `&& !isReplay` to the badge prompt condition.

### DE-2: `BadgeEarnedPrompt` shows "Submit Score" button when score already submitted [LOW]
- **File**: `app/page.tsx:795-809`
- **Description**: When `pieceCompleted = true` (score already submitted), the badge prompt still shows "Submit Score" as an action. Clicking it silently no-ops (`canSendOnChain = false`), leaving the user confused.
- **Trigger**: Complete all exercises → submit score → don't claim badge → replay exercise 5 → badge prompt re-fires with stale "Submit Score" button.

## State Leaks (subtle bugs)

### SL-1: `starDelta` computed independently in `handleMove` and `completeExercise` [MEDIUM]
- **File**: `app/page.tsx:420-423` vs `use-exercise-progress.ts:69`
- **Description**: Both `handleMove` and `completeExercise` independently call `computeStars()` with potentially different `currentExercise.optimalMoves` (see BF-1). The `newTotal` used to decide whether to show the badge prompt may differ from what `completeExercise` actually saves.
- **Impact**: Badge prompt could fire when threshold wasn't actually reached, or not fire when it was.

### SL-2: `loadProgress` doesn't validate individual star values [MEDIUM]
- **File**: `use-exercise-progress.ts:23-29`
- **Description**: localStorage validation checks `stars.length === 5` and `typeof exerciseIndex === "number"`, but doesn't validate individual star values. Corrupted localStorage like `stars: [99, -1, "foo", null, 3]` passes validation. `Math.max(99, newStars)` preserves the corruption. `totalStars` sums to absurd values, permanently breaking `badgeEarned`.
- **Trigger**: Manual localStorage edit or browser extension injecting bad data.

### SL-3: One-frame stale render on piece switch [INFO]
- **File**: `use-exercise-progress.ts:51-59`
- **Description**: When switching pieces, `progress` briefly holds default `{exerciseIndex: 0, stars: [0,0,0,0,0]}` before the `useEffect` rehydrates from localStorage. `currentExercise` resolves to exercise 0 of the new piece for one render frame.

## Race Conditions (timing bugs)

### RC-1: Timer fires while drawer/sheet is open [HIGH]
- **File**: `app/page.tsx:431-443, 451-453`
- **Description**: The `autoResetTimer` (1500ms for success, 1500ms for failure) doesn't check whether any sheet is open. If the user opens the exercise drawer after completing an exercise, the timer fires underneath — calling `advanceExercise()` + `resetBoard()` while the drawer is visible, desynchronizing the displayed exercise list from actual state.
- **Window**: 0-1500ms after exercise completion.
- **Fix**: Cancel timer when any sheet opens, or guard timer callback with `if (exerciseDrawerOpen) return`.

### RC-2: Multiple sheets can be open simultaneously [MEDIUM]
- **File**: `app/page.tsx:119-120,135`
- **Description**: `storeOpen`, `leaderboardOpen`, `badgeSheetOpen` are independent booleans. Adding `exerciseDrawerOpen` as a 4th independent state means the user could theoretically have multiple sheets open. Radix UI's overlay partially prevents this, but it's implicit behavior, not explicit code.
- **Fix**: Consider a single `activeSheet` state or closing other sheets when one opens.

## Edge Cases (unhandled)

### EC-1: `goToExercise` allows non-sequential star gaps [LOW]
- **File**: `use-exercise-progress.ts:100-102`
- **Description**: If localStorage is manually edited to have `stars: [0, 2, 0, 0, 0]`, `lastCompleted = 1`, `maxAllowed = 2`. User can navigate to exercise 2 despite exercise 0 having 0 stars. The system assumes stars are earned sequentially.

### EC-2: `handleBadgeEarnedDismiss` has dead code branch [LOW]
- **File**: `app/page.tsx:463-477`
- **Description**: The `if (!isLastExercise)` branch in `handleBadgeEarnedDismiss` never executes because badge earned only triggers when `isLastExercise === true`. The branch creates a false impression of auto-advance behavior.

### EC-3: `Exercise.id` field is never consumed [INFO]
- **File**: `lib/game/types.ts:26`
- **Description**: Every exercise has an `id` field (e.g., "rook-1") that is populated but never read by any component, hook, or test. Dead field. The spec's description table references these IDs but they're only useful if the drawer actually reads them.
